# 1.0 Readiness

Status: Draft
Owner: 0disoft

## Purpose

Define the evidence required before the package promises a stable `1.x` compatibility
surface. Feature count and implementation age are not release evidence.

## Required Gates

- Every exported function and type is documented in the public API contract and exercised
  through packed ESM, CJS, and TypeScript consumers.
- Accepted ADRs have implemented validation gates or explicitly documented residual platform
  limits.
- Bucketing, evaluation reasons, environment priority, flag schema, redaction defaults,
  audit-event shape, watcher behavior, and configuration-change key semantics have replay or
  contract coverage.
- Node.js 22 and 24 compatibility remains green on Linux, Windows, and macOS.
- At least one separate consumer repository has used a published prerelease through a normal
  install path, run real product behavior, and recorded an immutable revision and successful CI.
- Open high-impact correctness, privacy, packaging, or upgrade findings are resolved or block
  promotion.
- Migration notes identify every behavior change from the latest `0.x` release.

## Candidate Flow

1. Publish `1.0.0-rc.N` from the normal release workflow.
2. Keep prereleases on npm dist-tag `next`; they must not replace `latest`.
3. Install the exact registry artifact in ESM, CJS, CLI, watcher, audit, and OpenFeature event
   consumer paths.
4. Record cross-repository consumer feedback and any compatibility changes made during the candidate
   series. Use `.github/ISSUE_TEMPLATE/rc-consumer-report.md` so the report identifies an
   immutable consumer revision, successful CI run, exact package install path, environment,
   exercised surfaces, result, repository separation, and maintainer relationship.
5. After maintainer review, record the accepted report in
   `docs/testing/cross-repository-consumer-evidence.json`. A consumer in another repository may
   be maintained by the same owner when that relationship is disclosed; package-repository smoke
   alone cannot set the record to `accepted`.
6. Promote a new stable `1.0.0` artifact only after all required gates are satisfied. The
   `stable-release-gate` validation and release workflow reject stable `1.x` while the accepted
   evidence record is absent, so changing the patch version cannot bypass the gate. Do not retag
   or republish a candidate tarball as stable.

## Deferred Surfaces

Browser and multi-language SDKs are not `1.0` blockers. They require independent demand,
ownership, runtime policy, compatibility tests, and an ADR before implementation. The local
server package must not broaden its promise merely to make the major version look larger.

## Review Blockers

- A prerelease is published to npm `latest`.
- A stable release depends only on package-repository tests with no separate registry consumer result.
- A known compatibility change lacks migration guidance.
- Browser or another language is added without an owner and independent validation matrix.
